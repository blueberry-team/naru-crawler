# overview / tasks / techStack / benefitDetail 규칙

## 공통 원칙

- 명시된 내용만 사용한다.
- DOM 텍스트와 이미지 텍스트를 모두 읽는다.
- 같은 정보를 여러 필드에 중복 저장하지 않는다.
- 연수 / 온보딩 내용은 `overview` 에 넣지 말고 `targetCandidate.notes` 로 보낸다.

## overview

목적:

- 역할을 이해하는 데 필요한 설명을 저장한다.

포함 대상:

- 역할 설명
- mission
- 모집 배경
- 부서나 배치 맥락
- 일의 매력, 도전 요소, 영향

제외 대상:

- 연수 / 온보딩 흐름
- 긴 복지 나열
- 필수 / 우대 조건 나열

포맷:

- 하나의 문자열로 저장
- 단락 구분은 `\n`
- HTML 태그 사용 금지
- 과도한 빈 줄 금지
- 마지막 개행 금지

의미 있는 설명이 거의 없으면 `null`

## tasks

목적:

- 실제 담당 업무를 리스트로 저장한다.

우선 수집 위치:

1. `業務内容`, `仕事内容`, `担当業務`, `具体的な業務`, `ミッション` 아래 불릿
2. 본문 안 나열
3. `職務内容` 표 셀
4. 이미지 안 업무 내용

규칙:

- 불릿 1개 = 업무 1개
- 원문이 애매하면 애매함을 유지한다
- 구체 내용을 상상해서 넣지 않는다
- 가능하면 동작 중심 문장으로 다듬는다
- 중복 제거
- 명시가 없으면 `[]`
- 연수 / 온보딩 업무는 넣지 않는다

## techStack

아래 위치에서 명시된 기술명만 수집한다.

- `使用技術`
- `開発環境`
- `技術スタック`
- `利用ツール`
- 우대 요건 안의 명시적 기술 리스트
- 이미지 기반 스택 섹션

규칙:

- 기술명만 남긴다
- 설명 문구는 제거한다
- 확실할 때만 표기 정리
  - 예: `Type Script` → `TypeScript`
- 버전 정보가 명시돼 있으면 유지
  - 예: `Java 17`
- 없으면 `[]`

## benefitDetail

목적:

- 같은 공고 상세 페이지 안에 있는 간단한 복지 / 처우 정보를 저장한다.

수집 대상:

- `福利厚生`
- `待遇`
- `諸手当`
- `休暇`
- `制度`

규칙:

- 형식은 `Map<String, List<String>>`
- 가능하면 페이지 헤더를 key 로 사용
- 헤더가 없으면 `その他`
- 각 리스트 요소는 “원문에서 명확히 구분되는 의미 단위”를 유지한다
- 한 문장 안에서 여러 제도나 지원 내용이 묶여 설명돼 있으면 억지로 원자 단위로 쪼개지 않는다
- 서로 무관한 별도 문장이나 별도 항목을 임의로 하나로 합치지 않는다
- 원문에 없는 세부 제도명이나 하위 항목을 상상해서 분해하지 않는다
- 없으면 `{}`
- 회사 전체 복지 페이지까지 확장하지 않는다
- 이 내용을 `targetCandidate` 에 중복 저장하지 않는다
