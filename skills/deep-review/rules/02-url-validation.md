# URL 유효성 검증

## 대상 필드

`jobSourceUrl`

## 검증 순서

### 1단계: 접근 가능 여부

`jobSourceUrl` 에 **Playwright MCP**로 접근한다. Playwright가 불가능한 경우에만 WebFetch를 fallback으로 사용한다.

Playwright MCP 사용 시:
1. `browser_navigate` 로 URL 이동
2. 페이지 로딩 완료 대기
3. `browser_snapshot` 으로 DOM 스냅샷 확인
4. 勤務地, 給与 등 **개별 공고 섹션**만 확인 (회사 전체 정보 혼동 금지)

| 응답 | 처리 |
|------|------|
| 200 (정상 렌더링) | 2단계로 진행 |
| 3xx | 최종 도착 URL 확인 → 2단계로 진행 |
| 403 (WAF/Cloudflare) | Playwright로 재시도 → 여전히 차단이면 HOLD |
| 404 / 410 | HOLD — "원본 URL 접근 불가 (404)" |
| 5xx | HOLD — "원본 서버 오류 ({상태코드})" |
| 타임아웃 | 1회 재시도 → 여전히 타임아웃이면 HOLD |

### 2단계: 공고 상세 페이지 판별

200 응답을 받았더라도 아래를 확인한다.

**공고 상세 페이지가 맞는 경우:**
- 페이지에 직무 제목, 직무 내용, 응모 자격 등 개별 공고 고유 정보가 있다
- DB의 `title` 과 페이지의 직무 제목이 의미적으로 대응한다

**공고 상세가 아닌 경우 (HOLD):**
- 여러 공고가 나열된 목록/검색 결과 페이지 → HOLD "jobSourceUrl이 공고 상세가 아닌 채용 사이트 목록 페이지입니다"
- 채용 사이트 메인/탑 페이지 → HOLD "jobSourceUrl이 공고 상세가 아닌 채용 사이트 메인 페이지입니다"
- 지원서 작성 폼 페이지 → HOLD "jobSourceUrl이 지원서 페이지입니다. 공고 상세 URL로 수정 필요"
- 로그인/인증 필요 페이지 → HOLD "jobSourceUrl 접근에 로그인이 필요합니다"

### 3단계: title 일치 확인

페이지의 직무 제목과 DB의 `title` 을 대조한다.

- 완전 일치 또는 번역 관계에서 의미 일치 → PASS
- 전혀 다른 공고가 표시됨 → HOLD "원본 페이지의 공고와 적재된 title이 일치하지 않습니다"

### 리다이렉트 특수 처리

3xx 리다이렉트 후 최종 도착 URL 이 아래인 경우:
- 채용 메인/목록 페이지 → HOLD "리다이렉트 후 채용 메인으로 이동, 공고 만료 의심"
- 404 페이지 → HOLD "리다이렉트 후 404"
- 다른 공고 상세 → HOLD "리다이렉트 후 다른 공고로 이동, URL 매핑 오류 의심"
