# Edge Cases

이 문서는 중도채용 스킬에서 판단이 흔들리기 쉬운 케이스를 정리한 검토용 예시다.

각 케이스는 아래를 본다.

- 입력 상황
- 기대 처리
- 주의할 오답

## Case 1: `입사 시기 협의` 와 `가능한 한 빠르게` 구분

### 입력 상황 A

- 원문: `入社時期応相談`
- 원문: `入社日は相談の上決定`

### 기대 처리 A

- `joinDate = "입사 시기 협의"`

### 주의할 오답 A

- `가능한 한 빠르게`
- `미정`
- `협의 후 결정`

설명:

- 협의형 표현은 “즉시 채용 의지”와 다르다.
- 원문이 협의라고만 말하면 `입사 시기 협의`로 고정한다.

### 입력 상황 B

- 원문: `できるだけ早くご入社いただける方`
- 원문: `可能な限り早期にご入社いただける方`
- 원문: `早期入社歓迎`

### 기대 처리 B

- `joinDate = "가능한 한 빠르게"`

### 주의할 오답 B

- `입사 시기 협의`
- `즉시 입사`
- `미정`

설명:

- “빠를수록 좋다”는 의미이지 협의형 표현이 아니다.
- `即日` 같은 더 강한 표현이 없으면 `즉시 입사`로 올리지 않는다.

## Case 2: `즉시 입사` 와 `가능한 한 빠르게` 구분

### 입력 상황

- 원문 A: `即入社可`
- 원문 B: `即日入社可能`
- 원문 C: `決定後できるだけ早くご入社いただきます`

### 기대 처리

- A, B → `joinDate = "즉시 입사"`
- C → `joinDate = "내정 후 가능한 한 빠르게"`

### 주의할 오답

- 모든 표현을 `가능한 한 빠르게`로 통합
- 모든 표현을 `즉시 입사`로 통합

설명:

- `即日` / `即入社可` 는 즉시성의 강도가 더 높다.
- `決定後...` 는 내정 이후라는 조건이 있으므로 별도로 보존한다.

## Case 3: `리더 후보`, `매니저 후보` 같은 레벨 표현 처리

### 입력 상황 A

- 제목: `バックエンドエンジニア（リーダー候補）`
- 본문: `将来的にチームをリードいただくことを想定`

### 기대 처리 A

- `position = "BACKEND"`
- `title = "バックエンドエンジニア（リーダー候補）"` 또는 동등한 공식 직무명 유지
- `targetAudience = "경력직"`

### 주의할 오답 A

- `position = "LEADER"`
- `position = "BACKEND_LEADER"`
- `targetAudience = "백엔드 경력자(리더급)"`

설명:

- 레벨은 `position` 이 아니라 `title` / `targetAudience` / `targetCandidate` 로 보완한다.
- 직무는 백엔드고, 리더 후보는 보조 정보다.
- 다만 별도 연차 조건이 없다면 `targetAudience` 는 title 내용을 반복 입력하지 않고 기본값 `경력직` 으로 둔다.

### 입력 상황 B

- 제목: `プロダクトマネージャー（マネージャー候補）`
- 본문: `組織マネジメント経験歓迎`

### 기대 처리 B

- `position = "PM_PO"`
- `targetAudience = "경력직"`

### 주의할 오답 B

- `position = "MANAGER"`
- `position = "PM_MANAGER"`
- `targetAudience = "관리직"`

## Case 4: exact enum 이 없을 때 `*_OTHER` 사용 기준

### 입력 상황 A

- 제목: `ITエンジニア`
- 본문 전체를 봐도 백엔드 / 프론트엔드 / 인프라 / QA / 보안 구분이 없음

### 기대 처리 A

- `position = "IT_DEV_OTHER"`

### 주의할 오답 A

- `BACKEND`
- `FULLSTACK`
- `INFRA`

설명:

- 카테고리는 IT·개발로 보이지만 세부 직군 근거가 없으면 세부 enum으로 내려가지 않는다.

### 입력 상황 B

- 제목: `マーケティング担当`
- 본문에 광고 집행, 브랜드, CRM, SEO 중 무엇인지 특정할 근거가 없음

### 기대 처리 B

- `position = "MARKETING_AD_OTHER"`

### 주의할 오답 B

- `PERFORMANCE_MARKETING`
- `BRAND_MARKETING`
- `CRM_MARKETING`

### 입력 상황 C

- 제목: `営業職`
- 본문에 `法人` / `個人` / `代理店` 구분이 없음

### 기대 처리 C

- `position = "SALES_OTHER"`

### 주의할 오답 C

- `B2B_SALES`
- `B2C_SALES`
- `PARTNER_SALES`

## Case 5: 비슷한 직무명인데 세부 enum 으로 내려가야 하는 경우

### 입력 상황

- 제목: `サーバーサイドエンジニア`
- 본문: `Java / Spring Boot によるAPI開発`

### 기대 처리

- `position = "BACKEND"`

### 주의할 오답

- `IT_DEV_OTHER`

설명:

- 이 경우는 반대로 `*_OTHER` 를 쓰면 안 된다.
- 백엔드 근거가 충분하면 세부 enum을 써야 한다.

## Case 6: ATS에 상세 공고가 없고 지원 폼만 있는 경우

### 입력 상황

- 기업 사이트에서 `中途採用` 클릭
- 외부 ATS로 이동
- 이동한 페이지에는 로그인 / 지원 버튼만 있고
- 직무 설명, 자격 요건, 근무지, 급여, 전형 정보가 없음

### 기대 처리

- 이 ATS 폼 URL을 `jobSourceUrl` 로 쓰지 않는다
- 원래 사이트 또는 다른 상세 공고 페이지를 다시 찾는다
- 끝까지 상세 정보가 없으면 그 트랙은 제외한다

### 주의할 오답

- 지원 폼 URL을 `jobSourceUrl` 로 저장
- 상세 정보 없이 title만으로 payload 생성
- ATS 진입 자체를 상세 공고로 간주

## Case 7: ATS에 상세 공고는 있지만 지원 버튼도 함께 있는 경우

### 입력 상황

- 외부 ATS 페이지로 이동했지만
- 페이지 안에 `仕事内容`, `応募資格`, `勤務地`, `想定年収` 등 실제 상세 정보가 존재

### 기대 처리

- 해당 ATS 상세 페이지를 계속 사용한다
- `jobSourceUrl` 은 이 상세 ATS URL로 저장 가능

### 주의할 오답

- 외부 ATS라는 이유만으로 무조건 제외

설명:

- 기준은 도메인이 아니라 “상세 공고 정보가 실제로 있느냐”다.

## Case 8: `targetAudience` 에 너무 많은 정보를 넣고 싶어지는 경우

### 입력 상황

- 원문: `バックエンド開発経験3年以上、Java必須、AWS歓迎`

### 기대 처리

- `targetAudience = "백엔드 경력 3년 이상"`
- `mustHave` 에 Java 필수 조건 저장
- `niceToHave` 에 AWS 우대 조건 저장

### 주의할 오답

- `targetAudience = "백엔드 경력 3년 이상, Java 필수, AWS 우대"`

설명:

- `targetAudience` 는 카드 상단용 1줄 요약이다.
- 세부 조건은 `targetCandidate` 로 분리한다.

## Case 9: title 과 경력 연차 조건을 자연스럽게 분리하는 경우

### 입력 상황

- 제목: `バックエンドエンジニア（リーダー候補）`
- 본문: `実務経験3年以上`

### 기대 처리

- `position = "BACKEND"`
- `title = "バックエンドエンジニア（リーダー候補）"`
- `targetAudience = "경력 3년 이상"`

### 주의할 오답

- `targetAudience = "경력직"`
- `targetAudience = "백엔드 엔지니어(리더 후보)"`
- `title = "バックエンドエンジニア（リーダー候補） / 実務経験3年以上"`

설명:

- 직무명과 레벨은 `title` 에 남긴다.
- 별도 대상 조건인 연차는 `targetAudience` 로 분리한다.

## Case 10: 고용 형태가 title 에 포함된 경우

### 입력 상황

- 제목: `バックエンドエンジニア（正社員）`
- 다른 같은 페이지 항목: `バックエンドエンジニア（業務委託）`

### 기대 처리

- 공식 명칭 일부라면 title 에 유지 가능
- 단, `position` 은 둘 다 `BACKEND`

### 주의할 오답

- `position = "BACKEND_FULLTIME"`
- `position = "BACKEND_CONTRACT"`

설명:

- 고용 형태는 직무 enum이 아니라 title 또는 notes 쪽의 정보다.

## Case 11: 하나의 공고에 여러 근무 형태 표현이 섞인 경우

### 입력 상황

- 원문: `週3日出社を基本とし、その他の日は在宅勤務可`

### 기대 처리

- `workType = "HYBRID"`

### 주의할 오답

- `REMOTE`
- `ONSITE`

설명:

- 출사와 재택 병행 근거가 둘 다 명시되면 `HYBRID` 다.
