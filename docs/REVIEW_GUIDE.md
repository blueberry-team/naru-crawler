# 나루 공고 Deep Review 가이드 (25개 평가 기준)

30분마다 DRAFT 공고를 샘플링하여 다각도로 품질을 점검한다.
모든 검사 결과는 `history/` 폴더에 100건 단위로 기록한다.

---

## A. 연봉·보상 정확성 (3항목)

### A-1. 연봉 데이터 출처 확인
- 공고에 명시된 실제 연봉인지, 기업 평균 연봉 추정치인지, 아예 미기재인지 구분
- 출처를 명시: `공고 명시` / `기업 평균 추정` / `미기재` / `비공개(応相談)`
- 원본 페이지에서 연봉 관련 텍스트를 직접 발췌하여 비교

### A-2. 연봉 범위 일치 여부
- `salaryMin`/`salaryMax`가 원본 공고의 연봉 표기와 일치하는지
- 만엔 단위 ↔ 원 단위 환산 오류가 없는지
- 범위가 너무 넓거나(예: 300만~2000만) 비상식적이지 않은지

### A-3. 통화·단위 정확성
- 일본 엔(만엔) 기준인지 확인
- 월급/연봉/시급 혼동이 없는지
- `null`인 경우 원본에서도 정말 미기재인지 확인

---

## B. 콘텐츠 원본 충실도 (4항목)

### B-4. overview 원본 발췌 여부
- overview가 원본 공고에서 **실제 발췌·번역한 것인지** vs 캐싱/추정/자동생성인지 판별
- 원본 페이지의 직무 설명과 의미적으로 일치하는지 비교
- 원본에 없는 내용이 추가되거나, 핵심 내용이 누락되지 않았는지

### B-5. tasks 항목 원본 일치
- `tasks` 배열의 각 항목이 원본 JD의 "업무 내용"/"仕事内容" 섹션에서 추출된 것인지
- 원본에 있는 주요 업무가 빠지지 않았는지
- 원본에 없는 업무가 추가되지 않았는지

### B-6. targetCandidate 원본 일치
- `mustHave`(필수 자격)가 원본의 "必須条件"/"応募資格" 섹션과 일치하는지
- `niceToHave`(우대 사항)가 원본의 "歓迎条件"/"あれば尚可" 섹션과 일치하는지
- `workingConditions`(근무 조건)가 원본과 일치하는지

### B-7. selectionProcess 원본 일치
- 전형 절차가 원본의 "選考プロセス"/"選考フロー" 섹션과 일치하는지
- 면접 횟수, 서류 심사 유무 등이 정확한지

---

## C. 기업 특성 반영 (3항목)

### C-8. 업종·사업 특성 반영
- overview에 해당 기업이 어떤 산업에서 무엇을 하는 회사인지 맥락이 포함되었는지
- 예: "GMO미디어 주식회사는 GMO인터넷그룹 산하의 인터넷 미디어 기업입니다" — 적절한 맥락 제공

### C-9. 기업 규모·문화 차별화
- 대기업/스타트업/외국계 등 기업 특성이 overview에 드러나는지
- 복리후생·근무 환경 등 차별화 포인트가 원본에 있으면 반영되었는지

### C-10. 기업명 정확성
- 회사명이 정확한지 (자회사/관계사/지주회사 혼동 없는지)
- 예: "SEGA" vs "SEGA Fave" (별도 법인), "리크루트 홀딩스" vs "리크루트"
- `companySlug`가 올바른 기업을 가리키는지

---

## D. 분류 정확도 (4항목)

### D-11. position enum 정확성
- 직무 분류(`position`)가 실제 직무 내용과 일치하는지
- 예: "프론트엔드 엔지니어"가 `BACKEND`로 분류되면 ❌
- 매핑 가능한 enum이 없는 경우(DESIGNER, SALES_ENGINEER 등) 기록

### D-12. locations 정확성
- `locations` 배열이 실제 근무지와 일치하는지
- 원본에 오사카라고 적혀있는데 `TOKYO`로 매핑되면 ❌
- 매핑 불가 지역(Kawasaki, Nagoya, Hokkaido 등) 기록

### D-13. experienceLevel 정확성
- 경력 레벨이 원본의 자격 요건과 부합하는지
- "新卒" 공고가 `MID_CAREER`로 잡히면 ❌
- 원본에 "〇年以上" 등 경력 기준이 있으면 비교

### D-14. workType 정확성
- 근무 형태(출근/리모트/하이브리드)가 원본과 일치하는지
- "フルリモート"인데 `ONSITE`이면 ❌
- "ハイブリッド" 표기 여부 확인

---

## E. 데이터 완성도 (4항목)

### E-15. 필수 필드 누락 여부
- title, overview, tasks, position, locations, experienceLevel 모두 존재하는지
- 존재하지만 의미 없는 placeholder(예: "미정", "-") 가 아닌지

### E-16. techStack 정확성
- `techStack` 배열이 원본 공고의 기술 스택/사용 기술과 일치하는지
- 원본에 없는 기술이 추가되거나, 핵심 기술이 빠지지 않았는지

### E-17. deadline 정확성
- `deadline` 필드가 있으면 원본의 마감일과 일치하는지
- 마감일이 이미 지났는지
- 수시 채용인데 임의 deadline이 설정되진 않았는지

### E-18. joinDate 정확성
- 입사 시기가 원본과 일치하는지
- "即日", "随時", "응상담" 등이 정확히 반영되었는지

---

## F. 텍스트 품질 (3항목)

### F-19. HTML 잔여물·깨진 문자
- `<br>`, `&nbsp;`, `\n\n\n` 등 HTML/마크업 잔여물 없는지
- mojibake(깨진 문자) 없는지
- 인코딩 문제로 `?` 나 `□`가 표시되지 않는지

### F-20. 한국어 번역 자연스러움
- 일본어→한국어 번역이 자연스러운지
- 고유명사(회사명, 제품명, 기술명)가 적절히 처리되었는지
- 번역투가 과도하지 않은지

### F-21. 불필요한 보일러플레이트
- 기업의 일반 채용 안내문(모든 공고에 동일하게 들어가는 텍스트)이 overview에 과도하게 포함되지 않았는지
- 개별 공고의 고유 정보에 집중되어야 함

---

## G. URL·중복·현행성 (4항목)

### G-22. source_url 접근 가능
- `jobSourceUrl`에 HTTP GET 요청 시 200 응답인지
- 404/410 → 원본 삭제
- WAF 차단 → DB 데이터 양호 시 통과 가능 (별도 기록)

### G-23. 채용 종료 키워드
- 원본 페이지에 `募集終了`, `受付終了`, `クローズ`, `closed` 등 키워드 없는지
- 응모 버튼이 비활성화되어 있지 않은지

### G-24. 중복 공고 여부
- 동일 `jobSourceUrl`로 이미 적재된 공고가 있는지
- 제목이 거의 동일한 공고가 여러 개 있는지 (source_url은 다르지만 내용 중복)

### G-25. 공고 제목 의미 일치
- 원본 페이지의 제목과 나루에 적재된 `title`이 의미적으로 일치하는지
- 전혀 다른 공고가 표시되면 source_id 매핑 오류

## 평가 결과 기록 형식

### history/ 폴더 구조
```
history/
├── history001-100.md    # jobId 기준 1~100번 공고
├── history101-200.md    # 101~200번 공고
├── history201-300.md    # 201~300번 공고
└── ...
```

### 개별 공고 기록 형식
```markdown
## Job #645 — GMO미디어 | PHP 엔지니어

| # | 평가 항목 | 결과 | 코멘트 |
|---|----------|------|--------|
| A-1 | 연봉 출처 | ⚠️ | 미기재. 원본도 "スキル·経験に따라 결정" |
| A-2 | 연봉 범위 일치 | N/A | salaryMin/Max null |
| A-3 | 통화·단위 | N/A | |
| B-4 | overview 발췌 | ✅ | 원본 仕事内容에서 발췌 확인 |
| B-5 | tasks 일치 | ✅ | 4개 항목 원본과 일치 |
| ... | ... | ... | ... |
| G-25 | 제목 일치 | ✅ | 원본: "PHPエンジニア" → 한국어 번역 정확 |

**종합**: ✅ 22/25 통과, ⚠️ 2, ❌ 1
**판정**: PUBLISH 가능 (연봉 미기재는 원본도 동일)
```

## 리뷰 우선순위

1. **아직 한 번도 deep review 하지 않은 공고** 우선
2. **최근 적재된 공고** (새로 들어온 것부터)
3. **이전 리뷰에서 ⚠️ 주의였던 공고** 재검증
4. 이미 ✅ 정상이었던 공고도 주기적으로 재검증
