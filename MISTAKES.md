# 실수 기록 (Mistakes Log)

AI가 저지른 실수와 개선 사항을 기록한다. 매 적재 전 반드시 읽고 동일 실수를 반복하지 않는다.

---

## MISTAKE-001: 확인 전 테스트 데이터 삭제
- **상황**: DRAFT 테스트 공고 생성 후, 사용자가 어드민에서 확인하기 전에 바로 삭제
- **원인**: "테스트 데이터니까 삭제할게요"라고 독단적으로 판단
- **문제**: 사용자가 확인하려고 했는데 이미 삭제되어 검증 불가
- **교훈**: 사용자가 확인/승인하기 전에는 데이터를 삭제하거나 변경하지 말 것

---

## MISTAKE-002: fetch_job_detail이 잘못된 job 데이터 반환
- **상황**: 상세 API 호출 시 응답 JSON에서 job_id 일치 검사 없이 첫 항목 반환 → 모든 공고가 같은 데이터로 덮어씌워짐
- **해결**: 상세 API 호출 자체를 제거하고 목록 API 파라미터로 통합

## MISTAKE-003: 위치 오탐 (東京都 → KYOTO)
- **상황**: "東京都"가 "京都" 키워드와 substring 매칭 → TOKYO + KYOTO 동시 반환
- **해결**: 부분 매칭 방지 — 전체 지명(京都府, 京都市 등)으로 명시

## MISTAKE-004: HTML ruby 태그 기술스택 오탐
- **상황**: `<ruby>` 태그(일본어 후리가나)가 plain text 변환 후 "ruby" 남아 Ruby(언어)로 오탐
- **해결**: HTML 정리 시 `<ruby>`, `<rt>`, `<rp>` 태그 먼저 제거

---

## MISTAKE-005: 기업 slug 사전 확인 없이 새 회사 생성 → DB 오염
- **상황**: 이미 등록된 slug인데 확인 없이 새 회사 생성 → 동일 slug 중복, API 에러 유발
- **규칙**: 적재 전 반드시 `GET /api/companies?size=200`으로 기존 slug 확인

## MISTAKE-006: `status` vs `publishStatus` 필드명 혼동
- **상황**: `"status": "DRAFT"` 전송 → 서버에서 무시됨 → 기본값 PUBLISHED 적용, 공고 즉시 공개
- **규칙**: 반드시 `"publishStatus": "DRAFT"` 사용

## MISTAKE-007: 지원서 페이지 URL을 jobSourceUrl로 사용
- **상황**: 지원서 작성 페이지 URL을 공고 상세 URL로 착각하여 적재
- **규칙**: `jobSourceUrl`은 반드시 공고 상세 페이지 URL. 지원서, 목록 페이지 사용 금지

## MISTAKE-008: 제목에 불필요한 prefix 그대로 적재
- **상황**: 원문 제목의 번호, 코드 등을 제거하지 않고 그대로 사용
- **규칙**: prefix 제거 후 한국어 번역 적용

## MISTAKE-009: salaryMin/salaryMax null 방치
- **상황**: 원문에 "협의"라고만 써있어서 조사 없이 null로 적재
- **규칙**: OpenWork, 취업 에이전시 등 공개 데이터로 보수적 추정. 확신 없으면 salaryMin만 넣고 salaryMax는 null

---

## MISTAKE-010: 워크플로우 무시하고 무단 배치 적재
- **상황**: WORKFLOW.md를 읽지 않고 크롤러를 직접 실행 → 일본어 원문 그대로 배치 적재
- **위반 규칙**: 승인 없이 적재 / 일본어 원문 그대로 / 배치 적재 / WORKFLOW.md 미독
- **교훈**: "플로우대로"는 검토→승인→단건 플로우를 의미. 크롤러 자동 실행이 아님

## MISTAKE-011: 삭제 선언 후 바로 실행
- **상황**: "지금 삭제할게요"라고 선언 후 사용자 승인 없이 바로 삭제 실행
- **교훈**: 삭제는 항상 "삭제해도 될까요?" 물어본 후 OK 받고 실행

---

## MISTAKE-012: selectionProcess "복수 회" 어색한 표현
- **상황**: selectionProcess에 "면접 (복수 회)"라고 적음 → 부자연스러운 한국어
- **올바른 표현**: "서류 전형 → 면접 (2~3회) → 최종 합격" 또는 "서류 전형 → 면접 → 내정"
- **규칙**: selectionProcess는 자연스러운 한국어로 작성. "복수 회" 절대 사용 금지

---

## 앞으로 지켜야 할 원칙

1. **데이터 삭제는 반드시 사용자에게 먼저 물어볼 것** — 직접 실행 절대 금지
2. **수정은 사용자 허락 받은 후에만**
3. **추가는 승인 후 단건씩** — 배치 적재 금지
4. **품질 실패 공고도 로그에 남길 것** (silent fail 금지)
5. **DRY RUN 먼저, 보고, 승인, 실행** — 이 순서 절대 지킬 것
6. **모든 텍스트 한국어** — 일본어 원문 그대로 적재 금지
